# Mixed-Initiative Plan Review: Offline Reflective Improvement for Autonomous Agents

## Abstract

We describe a mixed-initiative system for improving the plans generated by
autonomous agents that use non-frontier language models for online planning.
The system combines an incremental online planner (which generates and executes
Python code blocks with embedded tool calls) with an offline review cycle in
which a human operator and a frontier-class language model collaboratively
critique, rewrite, and approve cached plans. Learnings from each review cycle
are injected into the agent's persistent world model, improving future online
planning without requiring changes to the planner itself. The approach treats
plan quality as a property that emerges from the interaction between online
execution, offline reflection, and accumulated institutional knowledge — rather
than from a single planning call.

## 1. Problem: Non-Frontier Models and Complex Task Execution

Autonomous agent systems that operate continuously — monitoring sensors,
managing goals, responding to user requests — face a fundamental resource
tension. Frontier language models (e.g., Claude Opus, GPT-4) produce
high-quality plans but are too expensive and slow for the high-frequency
orient/evaluate/decide cycles that a persistent agent requires. Smaller,
faster models (local models via vLLM/SGLang, or cheaper API-tier models via
OpenRouter) are economically viable for continuous operation but produce
measurably worse plans for complex, multi-step tasks.

Observed failure modes of non-frontier planners include:

- **Retry flailing.** When a tool call produces an unexpected result, the
  planner retries the same tool with slightly modified parameters rather than
  switching to a fundamentally different approach. We observed a planner call
  the `extract` tool five consecutive times with increasingly desperate
  instruction variations, eventually hardcoding the answer via string matching.

- **Inter-step data flow errors.** Plans consisting of multiple independent
  code blocks fail to pass intermediate results between steps. The planner
  treats each code block's `return` value as if it were accessible to the next
  step, when in fact the only cross-step communication channel is the infospace
  (persistent Notes created via tool calls).

- **Tool API misuse.** The planner calls tools that don't exist (`fs-write`),
  passes arguments in wrong formats, or fails to quote shell paths containing
  spaces. These errors are consistent across runs, suggesting the model's
  internal representation of the tool catalog is incomplete.

- **Loss of generality.** When recovering from failures, the planner
  hard-codes specific values observed during the current execution (file names,
  classification phrases) into the plan, producing plans that work for the
  current input but fail on any other.

These failure modes share a common structure: the planner makes a locally
reasonable decision that is globally suboptimal, and the online feedback loop
(compressed execution trace fed back to the planner) is insufficient to correct
it. The planner lacks the metacognitive capacity to recognize its own
antipatterns.

## 2. Architecture: The Online Planning Cycle

### 2.1 Incremental Code-Block Planning

The Cognitive Workbench planner (described in detail in
[incremental_planner_article.md](incremental_planner_article.md)) generates
plans as sequences of Python code blocks. Each code block may contain:

- Tool calls via `tool("tool-name", arg=value, out="$binding")` — these
  execute immediately during planning, producing real side effects.
- Data access via `get_text("$binding")` or `get_json("$binding")` — reading
  from the infospace (the agent's working memory of Notes and Collections).
- Arbitrary Python logic — loops, conditionals, string processing.
- Return statements that signal step completion, success, or failure.

Planning is incremental: after each code block executes, the planner observes
the result, reasons about progress (the THOUGHT phase), and decides whether
the goal is satisfied or what to do next. This produces a trace of
interleaved reasoning and execution.

### 2.2 Cached Plans and Replay

When a goal completes, the sequence of executed code blocks is cached as the
goal's `cached_plan_actions`. This cache serves two purposes:

1. **Replay.** Setting a goal's `execution_mode` to `"replay"` causes
   `/goal run` to execute the cached plan via `execute_plan_sync()` without
   invoking the planner. This is fast (seconds vs. minutes) and deterministic
   (same code blocks, same tool calls).

2. **Inspection.** The cached plan is the artifact that the offline review
   cycle operates on. It is a complete, executable record of what the planner
   decided to do and in what order.

### 2.3 Online Failure Recovery

The planner has limited online recovery capabilities:

- **Spiral detection.** A prompt instruction advises the planner to change
  approach after two consecutive failures of the same tool. In practice, this
  instruction is frequently ignored.
- **Spiral guard.** A structural enforcement mechanism in the executor blocks
  tool calls after two consecutive failures of the same tool, returning a
  synthetic error message: *"Tool 'X' blocked: 2 consecutive failures. Use a
  different tool or approach."* This forces the planner to adapt because its
  preferred tool literally stops responding.
- **Step budget.** The planner has a maximum step count (default 16). Complex
  tasks that consume steps on retries may exhaust the budget before completing.

These mechanisms prevent infinite loops but do not guarantee plan quality. A
plan that completes within budget may still contain redundant retry steps,
incorrect data flow, or overly specific logic.

## 3. The Offline Review Cycle

The core contribution of this work is a structured offline workflow in which a
human operator and a frontier language model collaboratively improve cached
plans, with learnings fed back into the agent's persistent knowledge store.

### 3.1 Review Bundle Generation

The command `/goal plan review <goal_id>` generates a review bundle consisting
of:

| Artifact | Contents |
|----------|----------|
| `goal_review_{id}.md` | Goal metadata, formatted plan, scheduler events, filtered execution log, automated structural checks |
| `goal_plan_{id}.py` | The cached plan in an editable format with step markers and metadata header |
| `goal_trace_{id}.txt` | Compressed planner trace (reasoning steps, tool calls, results) plus reflection frame |
| `plan_review_prompt.md` | Accumulated review guidelines and known antipatterns (shared across all goals) |

The structural checks are automated Level 1 analysis:
- Steps that returned `status: failed`
- Unknown action types encountered during execution
- Notes created with empty content
- HTTP errors from fetch operations
- Variables bound but never read in subsequent steps
- Debug `say` actions dumping raw content

### 3.2 Frontier-Model Review

The review bundle is presented to a frontier language model (in our case,
Claude Opus via Claude Code) along with the review prompt. The review prompt
is a self-documenting file that encodes:

1. **A review checklist** — goal alignment, inter-step data flow, tool API
   correctness, naming-vs-binding confusion, error handling, and generality
   preservation.
2. **Known antipatterns** — named, described, with fix instructions. These
   accumulate across review cycles.
3. **Revision history** — documenting when each pattern was discovered and
   from which goal's review.

The reviewer (frontier model + human oversight):

- Identifies which steps are retries vs. genuine pipeline stages
- Collapses retry steps into single steps with fallback branches
- Fixes tool API errors (quoting, nonexistent tools, wrong parameters)
- Corrects inter-step data flow (replacing return-value assumptions with
  explicit Note creation)
- Preserves generality by distinguishing instance parameters from strategy
  logic
- Removes hardcoded values from specific executions
- Adds a `### LEARNINGS` block to the plan file with actionable insights

### 3.3 Generality Preservation

A key tension in plan review is that the cached plan is an *instantiation* —
it contains specific file paths, variable names, and tool parameters from a
particular execution. The reviewer must improve the plan without losing the
generality implicit in the original goal text.

The review prompt encodes this as a checklist:

- **Instance parameters** (file paths from the goal text) — keep as-is.
- **Strategy logic** (the approach to the problem) — keep general.
- **Fallback strategies** (alternative approaches the planner tried) —
  consolidate as branches within a step, don't discard.
- **Hardcoded values** (literals from one execution) — replace with computed
  values.

The test question is: *"Would this plan still work if the goal text named a
different file?"*

### 3.4 Plan Commit and Learning Injection

The command `/goal plan commit <goal_id>` finalizes the review:

1. **Loads** the edited plan into the goal's `cached_plan_actions`.
2. **Sets** `execution_mode` to `"replay"`.
3. **Extracts** the `### LEARNINGS` block from the plan file.
4. **Classifies** each learning as either a general world-model fact or a
   tool-specific insight (based on whether it references a known tool name).
5. **Injects** learnings into the agent's persistent world model via the
   existing `WorldModel.update()` pathway, which applies Bayesian evidence
   tracking and generalization checks.
6. **Auto-consolidates** the world model if the planner-facing belief count
   exceeds a threshold, using a single LLM call that identifies duplicate or
   subsumed facts and merges their evidence counts into surviving entries.

The injected learnings become part of the `WORLD_MODEL: Facts:` context that
the online planner sees on every subsequent goal. This closes the loop:
offline review discovers patterns that online planning missed, and those
patterns are available to prevent the same mistakes in future online planning.

## 4. World Model as Institutional Memory

The agent's world model serves as institutional memory — a persistent store of
facts and tool contracts that accumulates across goal executions. It has two
layers:

- **Raw evidence store.** Every observation (from online `_reflect()` or
  offline `commit`) is recorded with support/contradiction counts, source
  attribution, and timestamps. This is the ground truth.
- **Beliefs cache.** Derived from raw evidence via Bayesian thresholds and
  recency weighting. This is what the planner sees. Low-confidence or stale
  facts are filtered out.

The offline review cycle contributes to the raw evidence store with `source:
"plan_review"` and `confidence: "high"`. Because review learnings are produced
by a frontier model with full context (goal text + execution trace + failure
analysis), they tend to be more specific and actionable than the online
`_reflect()` output, which is produced by the planner's own (weaker) model
with only a compressed trace.

### 4.1 Evidence-Preserving Consolidation

When the belief count exceeds a threshold (default: 30 facts), a consolidation
pass runs. This uses a single LLM call to identify duplicate or subsumed facts
and returns merge directives in the form `REMOVE n INTO m`. The surviving fact
absorbs the removed fact's evidence:

- `support_count` and `contradiction_count` are summed.
- `source_counts` are merged per-source.
- `first_observed_at` takes the earlier timestamp.
- `last_observed_at` takes the later timestamp.

This preserves Bayesian evidence history across consolidation events, avoiding
the information loss that would occur from simple deletion.

## 5. Evaluation

We evaluated the system on two goals from a knowledge management workflow:

### Goal 11: Clippings Processing Pipeline

A multi-step pipeline that reads a skip-list, identifies unprocessed files,
fetches content from source URLs, writes to an Obsidian vault, and updates the
skip-list.

| Metric | Initial (online) | After review |
|--------|-------------------|--------------|
| Steps | 17 (accumulated retries) | 3 |
| Key bug | Marked failed fetches as "processed" | Only marks successful fetches |
| Inter-step data flow | Broken (used return values) | Fixed (uses $newly_processed Note) |
| Replay time | ~3s | ~3s |

### Goal 12: Note Classification and Filing

Read Obsidian notes, classify subjects with a 1-3 word taxonomy phrase,
normalize, create subdirectories, and move files.

| Metric | Initial (online) | After review |
|--------|-------------------|--------------|
| Steps | 8 (6 retries + 2 verification) | 2 |
| LLM calls | 38 | N/A (replay) |
| Time | 257s | ~10s (replay) |
| Classification quality | Absurdly long phrases (>10 words) | 1-3 words (truncated) |
| Directory names | 100+ character paths | Clean normalized names |
| Shell quoting | Broken (spaces in filenames) | Correct (single-quoted) |
| Generality | Hardcoded file list | Dynamic discovery via fs-list |

Learnings injected from these two review cycles:

- `extract` tool: unreliable on Notes >10K chars; use preview.
- Always truncate classification output before use in filenames.
- Always single-quote paths in exec-script.
- Inter-step data flow requires explicit Note creation.
- Retry steps should be collapsed with fallback branches.

These learnings are now available to the online planner for all future goals.

## 6. Relationship to Prior Work

### Reflexion (Shinn et al., 2023)

Reflexion uses verbal self-reflection to improve agent performance across
episodes. The agent generates a reflection after each failed attempt, which
is stored in memory and included in future prompts. Our approach differs in
three ways:

1. **The reflector is a different (stronger) model.** Reflexion uses the same
   model for both action and reflection. We use the online planner's model for
   action and a frontier model for reflection. This sidesteps the fundamental
   limitation that a model cannot reliably identify its own systematic errors.

2. **Reflection operates on cached plans, not episodes.** Reflexion reflects
   on task outcomes. We reflect on plan *structure* — the code itself — which
   enables structural fixes (data flow, quoting, truncation) that outcome-level
   reflection would miss.

3. **Human-in-the-loop.** The review cycle is mixed-initiative: the frontier
   model proposes fixes, the human approves. This provides a quality gate that
   pure self-reflection lacks.

### LATS (Zhou et al., 2023) and Tree-of-Thought

Search-based approaches explore multiple plan paths at generation time. Our
approach is complementary: we explore a single path online, then improve it
offline. The cached plan is analogous to a single tree path that gets pruned
and refined post-hoc. The offline review has access to execution results that
tree search at generation time does not.

### Voyager (Wang et al., 2023)

Voyager builds a skill library from successful episodes. Our `cached_plan_actions`
serve a similar role — successfully reviewed plans can be replayed without
replanning. The key difference is that Voyager's skills are self-curated by
the acting model, while ours are curated by a frontier model with human
oversight.

### Meta-Learning and Learning to Plan

Traditional meta-learning approaches learn planning heuristics from many
episodes. Our world-model injection achieves a similar effect through a
different mechanism: rather than gradient updates, we inject natural-language
facts into the planner's context. This is more interpretable (facts can be
read and edited) and requires far fewer episodes (we observed meaningful
improvement from 2-3 review cycles).

## 7. Limitations and Future Work

**Manual triggering.** The review cycle requires the human to run
`/goal plan review`, invoke Claude Code, and run `/goal plan commit`. Full
automation would require the agent to recognize when a plan needs review
(e.g., based on error count or execution time) and initiate the cycle.

**Single-goal scope.** Each review cycle improves one goal's cached plan.
Cross-goal patterns (e.g., "all goals that use `extract` should use preview")
must be discovered manually from the review prompt's antipattern list.
Automated cross-goal analysis could identify recurring patterns.

**Consolidation quality.** World model consolidation uses the planner's model,
not a frontier model. Semantic equivalence judgments may be less accurate than
desired. However, the evidence-preserving merge means that a bad consolidation
(failing to identify a duplicate) merely leaves redundant facts — it does not
lose information.

**Generality validation.** We check for generality preservation during review
but do not automatically test it. Running the reviewed plan on different inputs
would validate that instance-specific fixes did not reduce generality.

**Planner prompt feedback.** Currently, world model facts are the only channel
from offline review to online planning. The review prompt's antipatterns
(which are more structured and actionable) are not directly injected into the
planner's system prompt. A future enhancement could select relevant
antipatterns based on the current goal and inject them as planning constraints.

## 8. Conclusion

The mixed-initiative plan review system addresses a practical gap in
autonomous agent architectures: the disconnect between what a cost-effective
online planner can produce and what complex tasks require. Rather than
attempting to make the online planner perfect (which would require a frontier
model at every planning step), we accept imperfect initial plans and improve
them through a structured offline cycle that leverages a frontier model's
superior analytical capabilities.

The key insight is that plan quality can be decomposed into *generation
quality* (online, cost-sensitive, good enough) and *curation quality*
(offline, thoroughness-sensitive, high standard). By separating these concerns
and connecting them through a persistent world model, the system achieves
improving plan quality over time without increasing the cost of online
operation.

The system is implemented in the Cognitive Workbench, an open-source
agent framework, and is available at
https://github.com/bdambrosio/Cognitive_workbench.
